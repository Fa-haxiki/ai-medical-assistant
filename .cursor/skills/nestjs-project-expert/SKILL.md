# NestJS 项目专家

## 使用场景

在以下场景应启用本技能：

- 用户在一个 NestJS 项目中开发或改动代码（例如 controller、service、module、guard、interceptor、pipe、filter 等）
- 用户提到“按项目现有结构来做”“遵守现在的规范”“照目前的模块划分扩展”等
- 涉及项目架构设计调整（模块拆分/合并、目录重构、跨模块依赖）

## 总体原则

1. **优先遵守项目现有结构与约定**
   - 先阅读相关 `module/controller/service` 文件，理解目前的分层和依赖关系
   - 新功能尽量放在已有模块内合适的层级；若必须新建模块，沿用现有命名与目录风格
   - 保持文件命名、类名、方法名、依赖注入方式与项目现有模式一致

2. **遇到不明确之处要先确认**
   - 对以下情况，不要直接拍脑袋实现，先向用户提出澄清问题：
     - 需求有多种合理设计方案（例如放在现有模块 A 还是新建模块 B）
     - 与当前代码风格/架构原则存在冲突
     - 需要做破坏性变更（接口字段变更、数据库结构变更等）
   - 提问时用简洁的要点式问题，并给出 1–2 个推荐方案供选择

3. **遵循 NestJS 及项目的常见最佳实践**
   - 控制器只负责 HTTP 层和简单参数处理，业务逻辑放到 service
   - 使用 DTO + 验证管道（class-validator / class-transformer 等）处理输入
   - 配置、常量、公共逻辑放到已有的 `config/common/shared` 等约定位置
   - 尽量通过依赖注入使用 service，而不是在文件内自行创建实例

## 具体工作流程

在帮助用户实现或修改功能时，遵循以下步骤：

1. **了解上下文**
   - 查找并阅读相关模块文件（如 `*.module.ts`、`*.controller.ts`、`*.service.ts`）
   - 如果用户提供了文件路径或片段，先基于这些内容推断项目结构和模式

2. **对齐设计与结构**
   - 判断该需求属于：
     - 在现有模块内增加 endpoint / method
     - 在现有模块内扩展 service 逻辑
     - 新建子模块/共享模块
   - 若存在多种合理放置位置或设计方式，先用简洁问题征询用户意见，再动手实现

3. **实现与改动**
   - 严格沿用既有命名和风格：
     - 文件命名（kebab-case）、类命名（PascalCase）、方法命名（camelCase）
     - 注入方式（构造函数注入）、装饰器（如 `@Injectable()`、`@Controller()` 等）
   - 保持职责清晰：
     - controller 处理路由、入参和出参
     - service 处理业务逻辑和数据访问（或调用 repository 层）
   - 若新增 DTO / 类型定义，放到项目中已有 DTO/类型的惯用位置

4. **必要时主动提示风险和折中**
   - 若改动会影响现有接口、数据库、队列、任务调度等，对可能的影响进行简要说明
   - 如需引入新依赖或新技术栈（例如新增缓存层、消息队列），说明理由与替代方案

5. **保持与用户对话**
   - 当对“项目规则”“目录结构”“模块边界”理解还不充分时，优先提问而不是武断决定
   - 提问尽量一轮收集尽可能多关键信息，减少多轮来回

## 提问模板示例

在不确定时，可以参考如下提问方式（可根据上下文简化）：

- “当前项目中，类似功能通常放在哪个模块？是继续放在现有的 `XModule` 里，还是需要新建模块？”
- “这个接口是扩展现有的 `GET /xxx` 还是需要新增一个单独的 endpoint？”
- “这个改动会改变现有响应结构，是否可以接受对前端造成 breaking change？”

## 输出风格

- 优先用 **简洁中文** 回答，必要时附上英文标识（如类名、装饰器）
- 给出修改建议时，尽量标明涉及到的文件名与类名，方便用户在项目中定位
- 代码示例要与当前项目结构、依赖、命名风格保持一致
